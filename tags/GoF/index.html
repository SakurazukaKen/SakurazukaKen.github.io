GoF 在前端开发中的应用

导语

再具体设计中应用设计模式能够方便我们对模型的理解与平时的交流, 并且经过前任大师们总结的模式能够有效提高架构的质量, 如今B/S广泛应用, 前端开发业务逻辑愈发复杂, 非常需要依赖模式来进行开发, 但是GoF的design pattern成书之时, web应用还没有广泛流行, 也无法讨论它总结的模式在前端开发中的有效性, 并且js的原型链继承方式无法与原来的oop想契合, 所以我在开发中尝试使用js改写实现了几个常用模式并探讨了其应用

Creational

Singleton模式

uml

code:

原来的设计中, 获取实例需要通过Instance()静态方法, 而在js中, 对象构造子即为函数, 可以在构造子中完成获取实例的操作.

应用

在前端中, 单例模式也是应用非常广泛的模式, spa需要缓存编译好的模板, 数据, 引用类库都是单例.

Factory模式

uml

code:

同样的, 利用js构造子的特性可以将FactoryMethod逻辑实现在构造中

应用:

前端中存在多种功能相似类型组件时, 经常会使用工厂模式, 比如一个scrollLayout工厂, 可以接受grid类型和list类型, 根据此来构造具有一个滚动方向的容器或者两个滚动方向的容器.

Prototype模式

uml

code:

Prototype模式与js的prototype继承不是一个概念

应用

用于实体对象的深复制, js原生的引用复制深复制, 使用原型模式来clone自身实现深复制方法

Behavior

Strategy模式

code

应用:

应用中经常需要动态的切换计算数据算法, 例如用户VIP等级不同导致商品折扣不同, 此时就可以应用策略模式

Observer模式

uml

code

应用

现代spa中模板绑定是必要功能, 而Observer模式很好地提供了监控数据修改并作出相应的解决方案, 我们项目的vue框架, 就通过劫持字段getter setter应用Observer模式实现了view的动态更新和数据依赖更新响应的问题

Structural

Bridge模式

uml

code

应用

桥接模式解决了多个抽象交叉导致无法复用的情况, 在前端中, 比较常见的场景是数据可以从服务器中获取也可以从本地获取, 处理后可以推送至服务器也可以保存至缓存, 通常写法只能是重复编写两种数据源推送至不同数据目的的方法, 应用桥接模式只需要实现两种抽象数据源和两种推送器

结

23种模式每个都是精华, 需要好好研究学习, 但是开发过程不可能一次遇到那么多问题, 多个模式每个都覆盖也不现实, 所以暂且取上述几个, 这篇文章就算作开篇, 日后遇到其他模式的场景在续文补充
